# Shopee Listing 自动出图 V2 领导汇报摘要

日期：2026-07-20  
当前阶段：飞书按钮、中转服务和 Coze 后台工作流已经完成真实 E2E。ID 12 经用户明确授权只重跑一次，Coze 运行 19 分 51 秒并完成 Listing 出图；飞书最终为“已完成”、1 张标准商品图和 7 张 Listing 图片。标准图最终字段已统一为短字段契约，生产分支已发布 `v0.0.12`：干净原图直接复用，需要处理的图片保存 `标准商品图生成` 节点结果。中转已增加标准图完成闸门和 OSS 持久化，健康检查及 OSS 无付费烟测通过。完成态重复请求返回 `duplicate=true`，未创建第二个 Coze 执行。竞品分支和自动成图质检暂缓。

## 一句话说明

这套工作流把运营上传的多张、角度不一、可能混有包装和场景的商品图，自动整理为可信商品身份与标准参考图，再按目标站点生成 6–8 张 1:1 本地化 Shopee Listing 图片，并以结构化结果返回。

## 核心业务链路

```text
多图与商品资料输入
→ 逐图识别与多图归并
→ 主商品判断和商品身份锁
→ 复用 / 抠图 / 语义提取标准参考图
→ 生成 6–8 张递进式详情页设计
→ 八站文案本地化与英文生图 Prompt 编译
→ 逐张生成详情图
→ 汇总标准图、详情图和任务状态
```

## 节点与解决的痛点

| 阶段 | 节点 | 主要职责 | 解决的业务痛点 |
| --- | --- | --- | --- |
| 输入 | 开始 | 接收商品名、多张自家图、商品资料、可选竞品图、平台和站点 | 统一运营输入；站点自动决定语言，不再重复填写目标语言 |
| 商品理解 | 逐图商品证据提取（批处理） | 逐张遍历商品图片 | 视觉模型一次接收多图不稳定；逐图分析便于定位哪张图有问题 |
| 商品理解 | 单图商品理解 | 区分商品、包装、说明书、配件、人物、背景和其他商品 | 避免把包装盒、说明书或场景道具当成主商品带入每张成图 |
| 商品理解 | 多图汇总与主商品判定 | 合并多角度、不同展示状态和同商品家族证据，选择主外观与互补角度图 | 解决只看一张图、不同角度被误判成不同商品、子 SKU 表面差异导致流程卡死的问题 |
| 商品理解 | 解析商品判定 JSON | 把模型输出拆成商品档案、身份锁、置信度、主图索引、互补图索引和标准图策略 | 将自然语言判断变成后续条件节点可稳定使用的结构化变量 |
| 质量闸门 | 主商品是否明确 | 依据 decision、confidence、主图索引和身份锁决定继续或补资料 | 主商品不清楚时在付费生图前停止，减少错误图片和无效成本 |
| 异常返回 | 组装待补资料结果 | 返回 needs_input 和具体补资料原因 | 运营知道要补什么，不会只看到技术报错或空结果 |
| 标准图 | 选择标准图源 | 根据主图索引选出标准参考图 URL | 从多图中自动定位最适合作为商品身份基准的一张图 |
| 标准图 | 标准图策略 | `reuse` 直接复用；`cutout/semantic_extract` 进入标准图生成 | 干净原图不额外付费处理，杂乱图保存处理后的标准参考图 |
| 标准图 | 复用标准图 URL | 对已干净、完整的商品图直接复用 | 避免不必要的抠图或生成，保留最真实的商品像素 |
| 标准图 | 组装标准商品图 Prompt | 为非 reuse 模式生成英文身份约束 Prompt | 从杂乱输入中整理完整商品，同时限制结构、数量和标识漂移 |
| 标准图 | 标准商品图生成 | 只在非 reuse 时执行 | 为需要处理的商品提供干净内部参考图 |
| 标准图 | 标准商品图 URL（变量聚合） | 取复用或生成分支的非空结果 | 将两路重新汇成一个稳定的 standard_product_image_url |
| 多参考图 | 组装详情图参考图数组 | 标准图加最多 3 张同一主外观的互补角度图 | 避免最终生图只看一个角度，也避免把包装、其他 SKU 或全部杂图无筛选传给模型 |
| 页面策划 | Shopee 详情页设计 V2 | 规划 6–8 张不同销售任务的 1:1 Listing 图片 | 解决每张图场景雷同、卖点重复、固定品类模板不通用、人物场景缺失的问题 |
| 页面策划 | 解析设计数组 | 把设计节点的 JSON 字符串解析为 Array<String> | 防止整组 6–8 条设计被当成一个 prompt，导致 JSON 失效或只生成一张图 |
| 批量生图 | 逐张详情图生成（批处理） | 每轮只处理一条设计，按顺序生成 6–8 张 | 实现真正批量生图，同时保持每张图的职责和顺序 |
| 本地化 | 本地化与生图 Prompt 编译 | 把中文设计稿转成英文画面指令，并将消费者文案改写为目标站点母语 | 解决机械翻译、当地表达不自然、内部标签被画进图片、国际生图模型对中文控制理解不稳定的问题 |
| 本地化 | 本地化与生图 Prompt 编译（身份约束） | 每张图重申商品身份、部件数量、连接拓扑和真实使用关系 | 降低颜色、结构、数量、使用姿态和不同 SKU 外观漂移 |
| 付费生图 | 详情图生成 | 使用精选参考图与单条 Prompt 生成 1:1 图片 | 将策划、本地化、商品身份和参考图真正转成可上架 Listing 图片；每轮一张、不同图不重复自动重试 |
| 结果 | 组装生成完成结果 | 汇总标准图 URL、详情图 URL、状态和说明 | 为飞书或其他系统提供固定、可读取的成功结果对象 |
| 结果 | 最终结果对象（变量聚合） | 汇合 completed 与 needs_input 两条分支 | 不需要多个结束节点，调用方只处理一种返回结构 |
| 结果 | 拆分最终结果 | 将最终对象拆成五个稳定字段 | 便于 Coze 结束节点和后续飞书中转服务直接绑定 |
| 结果 | 结束（唯一） | 输出 status、标准图、竞品摘要、详情图数组和 message | 建立稳定接口，后续接飞书或云服务器时不必再改 Coze 输出契约 |

## 已解决的关键问题

1. 支持 1–10 张杂乱商品图，不要求运营先人工选出唯一主图。
2. 能区分商品、包装、说明书、配件、人物和背景，减少错误元素进入成图。
3. 同商品家族的不同角度或子 SKU 不再轻易被判为冲突，同时最终只锁定一个主外观。
4. 抠图与生成式重建分开，普通抠图尽量保留真实商品像素。
5. 详情设计按 6–8 个购买决策任务递进，不再重复同一场景和卖点。
6. 覆盖 SG、MY、TH、VN、PH、ID、TW、BR 八站语言与生活场景。
7. 真人不是全局禁止项：需要穿戴、手持、接触身体或演示尺度时动态加入；危险或受管制商品不自动生成轻松人物场景。
8. 每轮只向生图模型传一条 String Prompt，解决整组数组塞入 prompt、JSON 非法和只生成一张的问题。
9. 付费生图前设置商品明确度闸门，且生图节点不自动重试，控制错误成本。

## 当前边界与下一阶段

- 生成模型对精确重复部件数量不能做到 100% 保证；身份锁、连接拓扑和可计数机位只能降低错误概率，当前仍需人工抽查数量关键商品。
- 当前生图插件返回 URL String，而 Coze 视觉理解输入要求 Image；自动成图一致性质检暂未接入。后续应接支持公网 URL 的视觉质检 API/插件，或可靠 URL→Image 节点。
- 首次真实 E2E 中，设计数组正常生成 6 项，但详情图批处理的第二轮在插件节点运行满 3 分钟后报 `plugin timeout`，剩余轮次未启动；插件字段 `asyn=true` 表示同步等待结果，但不会自动延长节点自身超时。`生成详情图` 节点随后设为 10 分钟并发布，保持自动重试为 0；第二次执行已验证该修正能支撑 22 分 34 秒的完整流程。
- 第二次真实 E2E 中，飞书约 2 秒完成异步受理，Coze 运行 22 分 34 秒后成功生成 7 张 Listing 图片；线上中转等待上限为 6 小时，因此此前失败与超时无关。中转已兼容大写 `Output` 字符串，并沿用同一执行记录完成回写；飞书现有 1 张标准商品图和 7 张 Listing 图片，状态为“已完成”。
- 本次旧执行的五字段结果里标准商品图 URL 为空，回收时使用同一执行记录中已成功生成的标准图节点结果，没有重新生图。根因是 `拆分最终结果` 的长输出变量名被 Coze 截断，代码返回键与声明不一致；正确配置是短字段 `standard_image_url`，再由结束节点映射回对外字段 `standard_product_image_url`。该修复曾发布为 `v0.0.6`，但后续发布重新引入了截断字段。
- ID 6、ID 7、ID 8 后续成功执行分别回写 7、7、5 张 Listing 图片，标准商品图均为 0。ID 8 的 `标准商品图生成`、`标准商品图 URL` 和结果组装节点均成功，只有 `拆分最终结果` 再次显示 `standard_product_ima`；因此这是最终变量传递回归，不是生图慢、飞书上传失败或中转轮询不足。
- ID 12 的授权回归执行 `7664243215327510538` 走完整原图复用分支，运行 19 分 51 秒后完成。节点级检查确认选择图源、复用 URL 和聚合绑定均有值；丢失发生在 `拆分最终结果`：代码返回键与声明的短输出名不一致。该记录已无付费复用原附件补回标准图，最终 1+7；字段修复已发布为 `v0.0.11`。
- 站点参数链路正常：ID 7、ID 8 以 `TH` 进入 Coze，ID 4 以 `VN` 进入 Coze，本地化节点也绑定了 `开始.zhandian`。浏览器运行页只渲染提示词当前可见区域，不能据此判断完整文本被删；用户提供的 432 行原文确认八站映射仍存在。为抑制英语漂移，完整原文已增加目标语言硬映射和输出前语言闸门。PH、SG 使用英语是设计行为；其他站点需使用增强后的完整提示词。
- 后续 ID 6、ID 7 的重叠执行分别失败在第二轮外部生图任务/Coze 批处理 `srcReg is null` 和 `标准商品图生成` 的 3 分钟 `plugin timeout`。ID 6 先成功生成 1 张，外部平台随后出现与该商品 Prompt 匹配的失败记录，但弹窗只显示尺寸与张数参数，尚无具体失败原因；ID 7 则确认标准图付费节点也必须设为 10 分钟。两条均不自动重跑。
- 首轮图片复盘发现：用户后来手工添加的实验抠图节点只保留了一个商品组件，并被错误绑定到 `标准商品图生成.image_urls`；前者输出 `Array<Image>`，后者要求 `Array<String>`。该实验节点现保持断开。`v0.0.12` 只在 `reuse` 时复用完整原图，`cutout/semantic_extract` 都保存标准商品图生成结果；标准图身份检查仍未接通，详情图多手部动作约束仍需人工抽查。
- 竞品分支已完成设计但第一版尚未启用；主流程稳定后再加入，只提取构图和销售策略，不复制竞品品牌、文案或外观。
- 飞书表格、“开始生成”按钮和云端中转服务已经落地；按钮受理、附件读取、Coze 上传、长时异步执行、1 张标准图与 7 张 Listing 回写以及完成态防重复计费均已验证。
- 第一版统一生成 1:1 Shopee Listing 图片，不再新增独立 3:4 生图节点。

## 对领导的建议表述
