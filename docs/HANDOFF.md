# 跨境电商 Listing 自动出图项目交接

更新日期：2026-07-20（Asia/Shanghai）

## 1. 一句话目标

让多个运营人员只在同一张飞书多维表格中上传商品资料、选择站点并点击“开始生成”，由一套云端服务统一调用 Coze 工作流，后台等待生图完成，再把标准商品图、6–8 张 Shopee Listing 图和处理状态写回原记录。

## 2. 当前阶段结论

### 已完成

- Coze V2 主流程已搭建并手工试运行通过。
- 支持一次上传 1–10 张同一商品、不同角度或不同子 SKU 的图片，并逐图提取证据后汇总商品身份。
- 支持标准图复用、普通抠图和必要时的语义提取策略。
- 支持 SG、MY、TH、VN、PH、ID、TW、BR 八个 Shopee 站点。
- 能生成 6–8 张 Listing 营销图；当前第一版统一为 `1:1`，标准商品参考图不计入这 6–8 张。

## 3. 多人运营的使用方式

运营人员只使用飞书，不直接进入 Coze，也不接触任何 Token：

1. 在飞书多维表格新建或填写一行任务。
2. 上传自家商品图，填写产品名称、产品资料、平台和站点。
3. 点击“开始生成”。
4. 查看任务状态：`待生成 → 处理中 → 已完成 / 待补充资料 / 失败`。
5. 在同一行下载“标准商品图”和“Listing 产出图”。

权限分层：

- 运营：飞书表格编辑权限，只能提交、查看和下载结果。
- 项目管理员：管理飞书字段、自动化和数据权限。
- 中转服务：持有一份飞书应用凭证和一份 Coze 服务访问令牌，代表系统完成调用。
- 运维：管理云服务器进程、端口和日志，不需要进入 Coze 工作流编辑器。

多人同时点击时，每一行记录都是独立任务。中转服务必须用 `record_id` 做幂等控制：同一行已经“处理中”或“已完成”时，不得再次提交付费生图。

## 4. 总体架构

~~~text
多个运营人员
    ↓
飞书多维表格按钮
    ↓  POST /jobs/feishu-listing（只传 record_id + 校验头）
云服务器中转服务 139.224.2.166:18082
    ├─ 读取飞书记录和下载附件
    ├─ 上传图片到 Coze
    ├─ 异步执行已发布的 Coze 工作流
    ├─ 保存 record_id ↔ execute_id
    ├─ 后台轮询 Coze 运行记录
    └─ 下载结果一次，同时归档 OSS、上传飞书附件并回写原记录
~~~

飞书自动化只负责提交任务，不能等待十几分钟的生图过程。轮询和回写必须由中转服务完成。

## 6. Coze 工作流契约

### 开始节点输入

| 变量 | 类型 | 必填 | 来源 |
| --- | --- | --- | --- |
| `product_name` | String | 是 | 飞书“产品名称” |
| `product_images` | Array<Image> | 是 | 飞书“自家产品图”上传到 Coze 后的文件 |
| `points` | String | 是 | 飞书“产品资料” |
| `competitor_images` | Array<Image> | 否 | 第一版固定 `[]` |
| `platform` | String | 是 | 固定 `Shopee` |
| `zhandian` | String | 是 | 飞书“站点” |

### 结束节点输出

| 变量 | 类型 | 说明 |
| --- | --- | --- |
| `status` | String | completed / needs_input / failed |
| `standard_product_image_url` | String | 标准商品参考图链接 |
| `competitor_summary` | String | 第一版固定“未启用竞品分析” |
| `result_image_urls` | Array<String> | 6–8 张 Listing 图片链接 |
| `message` | String | 结果或补资料原因 |

### 已确认的 Coze 配置要点

- `单图商品理解` 的视觉理解输入必须绑定批处理当前项 `item (in product_image)`，类型为真实 `Image`。
- 该节点的用户提示词中还必须显式引用 `{{current_image}}`；只写 `current_image` 会导致模型实际看不到图片。
- 条件节点固定文本写 `continue`，不要把双引号一起输入。
- 逐张详情图批处理的输入列表只能是 `design_list`。标准商品参考图数组作为普通上游变量传入批处理体，不能作为第二个批处理列表，否则会只运行一轮。
- 付费生图节点自动重试次数为 `0`，避免重复计费。
- 图片 Prompt 可以强调精确部件数量，但生成模型不能提供数学意义上的绝对保证；关键商品仍需抽检。
- 普通抠图优先使用非生成式 cutout，保留真实商品像素；只有 `semantic_extract` 才允许用生成模型整理标准参考图。

## 7. Coze 生产授权

本项目是“一个内部服务供多名运营共用”，不需要每个运营分别授权 Coze，也不需要创建 OAuth 普通应用。

在 Coze 中使用：`API 管理 → 授权 → 服务身份凭证 → 添加`。

服务访问令牌（SAT）配置：

- 只授权承载已发布 V2 工作流的空间。
- 至少开通工作流执行权限 `run`。
- 至少开通运行记录查询权限 `listRunHistory`。
- 还需开通界面中与“上传文件”对应的权限；以创建凭证时显示的实际权限名称为准，不凭空猜名称。
- Token 只写入服务器环境变量 `COZE_API_TOKEN`，不粘贴到聊天、文档或飞书。

个人访问令牌（PAT）只适合临时联调，不作为多人长期生产方案。

## 10. 中转服务最小实现

第一版已经实现以下能力：

1. `GET /health` 返回服务健康状态。
2. `POST /jobs/feishu-listing` 验证 `X-Bridge-Key` 和 `record_id`。
3. 读取飞书记录并校验必填字段。
4. 同一 `record_id` 若已处理中或已完成，直接返回已受理，不重复调用 Coze。
5. 将任务状态更新为“处理中”。
6. 下载飞书商品附件并上传到 Coze。
7. 异步执行已发布的 Coze 工作流，持久化 `record_id`、`execute_id`、状态和时间。
8. 后台每约 20 秒查询运行记录；不得因中间输出为 `null` 重新提交生图。
9. 成功后解析结束节点输出；标准商品图或 Listing 图片为空时拒绝误写“已完成”。
10. 将每个结果 URL 只下载一次，同一份字节先归档 OSS，再上传为飞书附件并回写。
11. 失败或超时只更新状态与错误说明，等待人工判断，不自动付费重跑。

第一版任务存储使用本地 SQLite 或等价的单机持久化即可，不部署 Redis、消息队列或独立数据库。单进程后台轮询足以完成当前目标。

当前中转代码与本地安全配置的 `BRIDGE_WORKERS` 均为 `4`，所以最多可同时处理 4 条工作流；更多已受理记录会在本进程线程池中等待。每条 Coze 工作流内部的详情图批处理并发保持 `1`，因此高峰时共享的图片模型 API Key 仍可能同时承受约 4 条生图调用。米核 AI 官网未公开固定并发数字；其官方前端错误说明包含 429“操作过于频繁，或同时使用人数太多”、图片 IPM/RPM 超限，以及“当前充值金额最大并发限制”，说明真实上限随账号充值档位、模型和上游容量变化，不能根据任务列表条数推断。未从米核官方取得本账号限额前，第一版运营上只允许 1 条记录处于正式生图阶段；保持 Coze 批处理并发 `1`、自动重试 `0`。若排队与生成总时长超过节点 600 秒，父工作流仍可能失败而外部任务继续运行，必须人工检查，禁止自动重提。
