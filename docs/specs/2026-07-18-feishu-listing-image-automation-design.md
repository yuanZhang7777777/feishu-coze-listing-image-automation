# 飞书多维表格 Listing 自动出图系统设计

日期：2026-07-18  
状态：飞书优先方案已确认；Shopee 内部运营 MVP 暂不启用竞品分支

## 1. 项目目标

让内部运营人员无需进入 Coze，只在飞书多维表格中完成：

1. 填写产品名称与真实商品资料。
2. 上传多张自家产品图。
3. 第一版不处理竞品参考图；该能力在主流程稳定后再启用。
4. 选择平台与站点；目标语言由站点自动映射。
5. 点击开始生成。
6. 在同一行查看标准商品图、竞品分析、任务状态和最终 Listing 图片。

第一版只跑通 Shopee 详情页图片。TikTok Shop 和其他平台在阶段五之后分别增加独立模板，不共用 Shopee Prompt 硬套。

## 2. 最小系统架构

~~~text
飞书多维表格
  → 点击“开始生成”
  → 飞书自动化发送 HTTP 请求
  → 轻量中转服务
  → 下载飞书附件并上传给 Coze
  → 调用已发布的 Coze 工作流
  → 获取标准商品图、竞品策略和 Listing 图片
  → 将状态、说明和图片回写原记录
~~~

飞书是运营界面，Coze 是 AI 引擎，中转服务只负责附件、鉴权、异步任务和结果回写。中转服务不需要开发独立前端。

## 3. 第一次创建飞书多维表格

### 3.1 创建文件

1. 打开飞书客户端或飞书网页版。
2. 进入“云文档”。
3. 点击“新建”。
4. 选择“多维表格”。
5. 选择“从空白新建”。
6. 文件命名为“跨境电商 Listing 自动出图中心”。
7. 将默认数据表重命名为“出图任务”。

不同版本的入口文字可能略有变化；目标是创建独立的多维表格文件，不是在普通电子表格单元格里插入多维表格模块。

### 3.2 创建字段

第一列直接重命名为“产品名称”，作为每条任务的主字段。

| 顺序 | 字段名 | 飞书字段类型 | 运营是否填写 | 规则 |
| --- | --- | --- | --- | --- |
| 1 | 产品名称 | 单行文本 | 必填 | 用于识别混合图片中的主商品 |
| 2 | 自家产品图 | 附件 | 必填 | 每行建议上传 1–10 张；允许白底图、场景图、细节图和包装混拍 |
| 3 | 产品资料 | 多行文本 | 必填 | 商品事实、卖点、规格、结构数量、包装清单和使用限制 |
| 4 | 竞品参考图 | 附件 | 第一版不填写 | 字段可保留并隐藏；第一版中转服务不读取、不上传、不传入生图模型 |
| 5 | 平台 | 单选 | 必填 | 第一版只设置 Shopee |
| 6 | 站点 | 单选 | 必填 | SG、MY、TH、VN、PH、ID、TW、BR |
| 7 | 开始生成 | 按钮 | 运营点击 | 触发当前行自动化 |
| 8 | 任务状态 | 单选 | 系统回写 | 草稿、待生成、处理中、已完成、待补充资料、失败 |
| 9 | 标准商品图 | 附件 | 系统回写 | `reuse` 时为选中的干净原图；需要处理时为标准商品图生成节点输出 |
| 10 | 竞品分析摘要 | 多行文本 | 系统回写 | 可复用的分屏、构图、场景与文案策略 |
| 11 | Listing 产出图 | 附件 | 系统回写 | 最终 6–8 张 Shopee 详情页图 |
| 12 | 处理说明 | 多行文本 | 系统回写 | 补资料原因或技术错误 |
| 13 | 最后更新时间 | 最后更新时间 | 自动 | 用于排查任务是否卡住 |

第一版不要增加模型选择、分辨率、并发数、Prompt 版本、计费金额等运营字段。模型与参数先由工作流统一管理。

站点与语言固定映射：SG 使用英语，MY 使用马来西亚马来语，TH 使用泰语，VN 使用越南语，PH 使用英语，ID 使用印度尼西亚语，TW 使用繁体中文，BR 使用巴西葡萄牙语。第一版不设置“目标语言”字段，避免运营重复选择或站点与语言冲突。

如果表中保留了人工编号字段“id”，它只用于运营查看，不能代替飞书内部 `record_id`。自动化调用中转服务时必须引用触发记录的系统记录 ID。

## 4. 运营录入方式

第一版由内部运营直接在“出图任务”表格中填写，不创建提交表单。一次只填写一个商品和一个站点；产品图可以上传多张；多款混拍时必须在产品名称中写清具体款式。竞品参考图字段暂时隐藏或留空，不参与第一版任务。

当外部人员、移动端用户或无表格编辑权限的人员需要提交任务时，再增加表单视图。

## 5. 运营视图

第一版只保留默认表格视图，不额外创建状态视图。任务数量增加后，再按“处理中、待补充资料、已完成、失败”创建筛选视图；这些视图只影响查看方式，不参与工作流触发。

## 6. 创建飞书自动化

### 6.1 操作入口

1. 打开多维表格。
2. 点击右上角“自动化”。
3. 点击“创建自动化流程”。
4. 触发条件选择“点击按钮时”。
5. 按钮字段选择“开始生成”。

飞书也支持“新增或修改记录满足条件时”自动触发，但第一版使用按钮更安全，能避免运营上传图片过程中重复触发。

### 6.2 自动化动作

按顺序配置：

1. 修改当前记录：
   - 任务状态 = 待生成
   - 处理说明 = 已提交，等待系统处理
2. 发送 HTTP 请求：
   - 方法：POST
   - URL：阶段三部署的中转服务地址
   - 请求头：只放中转服务自定义校验密钥，不放飞书或 Coze API Key
   - 请求体：只传触发记录的系统 `record_id`；产品名称、平台、站点和附件均由中转服务通过飞书 API 读取

自家产品图和竞品参考图不直接塞进 HTTP JSON。中转服务拿 record_id 后，通过飞书 API 读取当前记录并下载附件。

自动化保存后先保持关闭，等中转服务和 Coze 工作流准备完成再启用。

## 7. 状态流转

~~~text
草稿
  → 点击开始生成
  → 待生成
  → 处理中
  → 已完成

处理中
  → 待补充资料

处理中
  → 失败

待补充资料
  → 运营修改资料并再次点击开始生成
  → 待生成
~~~

“待补充资料”用于业务输入不足，例如：

- 无法确认哪一个是主商品。
- 多个款式混在一起但产品名称不明确。
- 商品只在包装印刷图上出现。
- 所有图片都严重模糊或遮挡。
- 缺少影响商品结构的必要事实。

此时：

- Listing 产出图保持为空。
- 任务状态写“待补充资料”。
- 处理说明写清需要补什么。

“失败”只用于技术故障，例如飞书附件下载失败、Coze 调用失败或图片结果无法上传回飞书。

## 8. 中转服务职责

### 8.0 线上服务器网络基线

2026-07-21 迁移后直接使用 `http://139.224.2.166:18082`，不配置域名、TLS 证书或 Nginx。本项目固定使用 `18082`，不得监听、转发或修改服务器上的其他服务端口：

- 公网入站：云安全组现有 IPv4 TCP `18000-19000` 规则覆盖 `18082`；中转服务本身只监听 `18082`，不使用该范围内其他端口。
- 运维入站：保留服务器现有 SSH `2323` 端口和访问策略，不另开 SSH 端口，不修改其他服务端口。
- 出站：允许 TCP `443`、DNS `53` 和时间同步；服务需要访问飞书开放平台、Coze API、Coze 文件服务以及生成图片实际所在的 HTTPS 域名。
- 不开放数据库、Redis 或其他中间件端口；第一版使用服务本地轻量任务存储。

应用监听 `0.0.0.0:18082`，提供 `/health` 和 `/jobs/feishu-listing` 两个路径。第一版使用服务器 Python 3.12 虚拟环境和 systemd。由于 HTTP 不加密，请求正文只传 `record_id`，校验头使用独立随机长密钥；飞书 App Secret、Coze Token 等真实密钥只放权限为 `640`、归属 `root:listingbridge` 的服务器环境文件。正式给运营长期使用前再升级 HTTPS。

### 8.1 飞书 API 访问准备

第一版只服务当前企业内部运营，因此使用“企业自建应用”，不创建商店应用、机器人、网页应用或多维表格插件。

接入步骤：

1. 在飞书开放平台创建企业自建应用，名称可用“Listing 自动出图中转”。
2. 在应用权限管理中申请“查看、评论、编辑和管理多维表格”。该权限可覆盖记录读写以及当前多维表格素材上传、下载所需的接口权限；仍需确保应用拥有目标多维表格本身的文档权限。
3. 发布并在当前企业内启用该应用。
4. 在目标多维表格的“更多 → 添加文档应用”中加入该应用；如果表格开启高级权限，为应用授予“可管理”。
5. 从正式多维表格 URL 或 API 获取 `app_token` 和 `table_id`。共享视图 URL 中以 `shrc` 开头的分享 token 不能代替这两个标识。

中转服务固定保存：

- `FEISHU_APP_ID`：应用标识，不作为业务字段。
- `FEISHU_APP_SECRET`：机密，只存环境变量，不在聊天、多维表格或代码中出现。
- `FEISHU_APP_TOKEN`：目标多维表格标识。
- `FEISHU_TABLE_ID`：目标“出图任务”数据表标识。

`view_id` 第一版不需要。`record_id` 也不是固定配置，而是每次点击“开始生成”时，由飞书自动化把当前触发记录的系统 ID 传给中转服务。

中转服务只保留一个入口，例如：

~~~text
POST /jobs/feishu-listing
~~~

处理顺序：

1. 验证飞书自动化携带的自定义密钥。
2. 根据 record_id 读取飞书当前记录。
3. 如果任务状态已经是“处理中”或“已完成”，直接忽略重复请求。
4. 将任务状态改为“处理中”。
5. 下载“自家产品图”附件；第一版不读取竞品参考图。
6. 将图片上传给 Coze，得到工作流可用的文件标识。
7. 以异步方式调用已发布的 Coze 工作流，立即保存返回的 `execute_id`；飞书 HTTP 自动化此时只接收“已受理”，不等待生图完成。
8. 中转服务在后台按 `execute_id` 查询 Coze 工作流执行记录。`Running` 时继续等待，`Success` 时读取最终 `output`，`Fail` 时记录错误并停止。
9. 获取标准商品图、竞品分析摘要、Listing 图片和处理信息。
10. 将结果 URL 只下载一次；同一份字节先归档 OSS，再上传到飞书多维表格素材空间。
11. 标准商品图或 Listing 图片为空时拒绝写“已完成”；齐全时再将附件 token、摘要和状态回写原记录。

异步调用刚返回时，业务 `data` 或图片字段为 `null` 并不代表失败；只要同时取得非空 `execute_id`，就表示任务已受理，后续结果必须从执行记录中读取。若执行记录最终为 `Success`，但工作流最终 `output.result_image_urls` 仍为空，才按生图节点输出异常处理。不得因中间结果为 `null` 自动重新提交付费生图任务。

中转服务不要求 URL 立即返回：它最长轮询既有 `execute_id` 6 小时，并只在工作流进入终态后解析结果。工作流已经是 `Success` 时，最终输出是终态快照；缺字段不会因继续等待而补齐，不能重新提交生图。2026-07-19 的历史标准图空字段均来自最终字段契约回归；ID 12 回归确认上游复用 URL 和聚合值非空，真正问题是拆分代码返回 `standard_product_image_url`、节点却只声明 `standard_image_url`。该错配已修复。2026-07-20 的中转版本进一步把标准图设为完成态必填字段，避免以后再次出现“已完成但没有标准图”。

生成图片供应商的 URL 可能短时有效。中转服务在 Coze 成功后立刻下载标准图和 Listing 图片，并将相同字节同时写入 Aliyun OSS 与飞书附件；OSS 对象键按 `前缀/record_id/execute_id/standard|listing/序号-文件名` 组织。归档失败时整条任务不得误写“已完成”。`GET /health` 的 `oss_configured=true` 表示 OSS 配置完整，但不代替对象上传烟测。

若图片服务提供正式的异步提交接口与独立任务查询接口，后续可由中转服务持久化每张图的外部 `task_id` 并直接轮询，这样不依赖 Coze 插件节点长时间等待。当前插件声明 `data.id`/`data.url`，但 ID 12 显示节点成功并不保证 URL 非空；中转服务最终仍只有 Coze `execute_id`，不能从终态结果取得可查询的外部任务 ID。实施外部直轮询前必须取得服务商正式的提交、查询、终态、错误和结果 URL 契约，不反向猜测网页私有接口。

真实异步运行还确认，Coze 运行记录的 `output` 可能解析为 `{"Output":"<五字段 JSON 字符串>","node_status":"{}"}`。中转服务必须兼容大写 `Output` 包装并再次解析其中 JSON；不能因顶层没有 `result_image_urls` 就判定图片为空。2026-07-19 第二次 ID 3 运行已成功生成 7 张 Listing 图片；兼容修复部署后，已沿用原执行记录完成回写，没有重新付费生图。

当前中转实现调用 Coze 时只提供固定 `workflow_id`、业务参数和 `is_async=true`，没有指定或保存工作流版本号。因此飞书“开始生成”按钮运行该 `workflow_id` 的最新已发布版本；Coze 编辑器中发布后的草稿修改不会被按钮执行，只有再次发布后才生效。

轮询由中转服务完成，不由飞书多维表格自动化完成。第一版每 20 秒查询一次即可，并设置可配置的业务等待上限；超过上限只把任务标记为“失败/超时待人工检查”，不自动重新生图。飞书记录只承担状态展示：`待生成 → 处理中 → 已完成 / 待补充资料 / 失败`。

中转服务的 `BRIDGE_WORKERS` 默认及当前本地安全配置均为 `4`，所以最多可同时执行 4 条工作流；Coze 内部逐张详情图批处理并发保持 `1`。共享图片模型 API Key 的实际并发、RPM/IPM 和队列上限由供应商账号档位与上游容量决定。米核 AI 官网未公开固定数字，其官方前端错误说明显示可能出现 429、RPM/IPM 超限、上游负载饱和，以及“当前充值金额最大并发限制”。因此任务列表数量不能用来推断并发值。官方未确认本账号限额前，第一版运营只允许 1 条记录处于正式生图阶段；不要通过并发压测消耗付费额度，也不要因 429、排队或超时自动重提。

所有密钥只保存在中转服务环境变量中：

- 飞书 App ID
- 飞书 App Secret
- Coze 服务访问令牌（SAT，生产使用）
- Coze Workflow ID
- 飞书自动化与中转服务之间的校验密钥

不得把任何 API Key 写入多维表格字段、Coze 普通输入变量或飞书 HTTP 请求正文。

多人使用时，运营人员只需要目标多维表格的飞书编辑权限，不需要各自创建 Coze 账号、OAuth 应用或 Token。中转服务使用一份服务身份统一调用已发布的 Coze 工作流：

- 正式环境使用 Coze 服务访问令牌（SAT），授权范围只选择承载该工作流的空间。
- 权限至少覆盖工作流执行、工作流运行记录查询，以及把飞书附件上传为 Coze 工作流文件所需的文件上传能力。
- 个人访问令牌（PAT）最长有效期有限，只适合短期联调，不作为多人长期使用方案。
- 当前 API 管理中的“创建 OAuth 应用”页面不是本项目生产接入入口。

## 9. Coze 工作流输入输出契约

### 9.1 开始节点输入

| 变量 | 类型 | 来源 |
| --- | --- | --- |
| product_name | String | 飞书产品名称 |
| product_images | Array<Image/File> | 飞书自家产品图 |
| points | String | 飞书产品资料 |
| competitor_images | Array<Image/File> | 第一版固定传空数组 `[]`，保留接口便于后续启用 |
| platform | String | 第一版固定 Shopee |
| zhandian | String | 飞书站点 |

语言本地化节点不接收独立语言输入，只根据 `zhandian` 使用固定语言映射：SG→英语、MY→马来西亚马来语、TH→泰语、VN→越南语、PH→英语、ID→印度尼西亚语、TW→繁体中文、BR→巴西葡萄牙语。

### 9.2 节点顺序

~~~text
开始
→ 自家多图商品理解
→ 商品图标准化策略判断

→ 生图插件
→ 汇总
→ 结束
~~~

生产版本 `v0.0.12` 的标准图条件为：`reuse` 直接复用选中的完整原图；其他模式进入 `组装标准商品图 Prompt → 标准商品图生成`，并把该节点结果作为标准图。用户后加的 `按索引抠图` 实验节点仍保持断开，不得绑定到 `标准商品图生成.image_urls`；后者只绑定 `选择标准图源.source_image_urls`。如果以后重新启用非生成式抠图，必须先单独验证完整主体和 URL 契约，再修改生产分支。

## 10. 防止重复运行与重复计费

- 运营只通过“开始生成”按钮触发。
- 中转服务按 record_id 检查当前任务状态。
- 处理中和已完成的任务拒绝重复执行。
- 每次正式运行生成唯一 job_id，并记录在服务日志中。
- `record_id` 是幂等键，`execute_id` 是 Coze 异步执行追踪标识；两者必须在中转服务日志或任务存储中关联。
- 付费生图节点不做自动重试。
- Coze `标准商品图生成` 与详情图插件节点自身的超时时间都设为 10 分钟；插件字段 `asyn=true` 表示同步等待结果，不会自动覆盖默认 3 分钟节点超时。
- 飞书自动化 HTTP 请求只负责提交任务，不等待全部图片生成完成。
- 中转服务只轮询既有 `execute_id` 的执行状态，不重复调用工作流提交接口。
- 技术失败由运营确认后重新点击，不自动无限重跑。

## 11. 第一次试运行

不要一开始批量跑几十行。先准备五条测试记录：

1. 一张干净白底商品图。
2. 商品、包装盒、说明书和配件混拍。
3. 多张不同角度的同一商品。
4. 人物正在使用商品的场景图。
5. 同商品家族多个颜色/花纹子 SKU 分别提供不同角度；另加一组核心结构或用途实质不同的多个商品且产品名称故意写得不明确。

验收结果：

- 前四条能选择正确主商品并生成标准商品图。
- 同商品家族多子 SKU 应继续并选择主 SKU；只有核心结构或用途实质不同且无法定位目标的测试组必须进入“待补充资料”。
- 竞品 Logo、文字和产品外观没有进入最终图片。
- 输入可跨同商品家族的颜色、花纹和角度取证；最终一套生成图必须保持选定主 SKU 的颜色、花纹、结构、Logo 和关键数量，不得跨 SKU 混合外观。
- 同一行只产生一次付费生成任务。
- 已完成结果能回写原记录。

## 12. 实施顺序

### 阶段一：飞书表格

创建多维表格、字段和未启用的按钮自动化；第一版不创建提交表单和额外状态视图。

### 阶段三：中转服务

实现飞书附件下载、Coze 文件上传、异步任务调用和飞书结果回写。

### 阶段四：单行联调

只启用一条按钮自动化，用一行任务完成端到端验证。

### 阶段五：运营试用

开放给少量运营人员，观察补资料率、失败率、单商品成本和图片可用率，再决定是否增加 TikTok Shop 模板或独立平台。
